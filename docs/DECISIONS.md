# 설계 결정 기록 (Decision Log)

중요한 설계·제품 결정을 기록한다. "왜 이렇게 했더라?"에 답하기 위한 문서다.

각 결정은 다음 구조를 따른다:
- **상황**: 어떤 논의에서 나왔는가
- **결정**: 무엇을 하기로 했는가
- **근거**: 왜 그런 결정을 내렸는가
- **재고 조건**: 어떤 상황이 되면 다시 검토할 수 있는가
- **일자**

## 1. 자동매매 기능 구현하지 않음

**상황**: 초기 설계 논의 중 사용자가 "시그널 감지 시 증권사 API로 자동 주문 집행" 기능을 요청.

**결정**: 구현하지 않는다. 대신 아래 세 가지 반자동 편의 기능으로 대체한다.
- 증권사 앱 딥링크 (주문 화면까지 이동, 체결은 수동) — F19
- (향후) 페이퍼 트레이딩 모드 — 가상 자금 전략 검증
- (향후) 시그널 백테스트 — 점수 체계 유효성 통계 검증

**근거**:

1. **시그널 품질이 검증되지 않음.** `SIGNALS.md`의 점수 체계는 "일반적 해석" 기반 휴리스틱이지, 실제 수익을 증명한 알파가 아니다. 같은 공시 유형이 상승·하락 양방향으로 작동하는 사례가 많다. 검증 없이 돈을 맡기는 것은 동전 던지기에 가까울 수 있다.

2. **원래 문제의식과 자기모순.** 시스템의 출발점은 "기사는 이미 늦은 정보"였다. 뉴스·공시 시점에 자동 매매를 집행하면 이미 시장이 반응한 뒤 불리한 가격에 진입하게 된다. 호재성 공시 대부분은 장 시작부터 갭 상승한 채 거래된다. 자동매매는 이 구조적 약점을 고착화한다.

3. **백테스트·리스크 관리 부재.** 포지션 사이징, 손절 규칙, 상관관계 관리, 슬리피지·수수료 계산 — 자동매매 시스템의 핵심인 이 관리 레이어가 전혀 설계돼 있지 않다. 매매 로직보다 이 부분이 수익률에 미치는 영향이 크다.

4. **LLM 할루시네이션 × 자동 실행.** Week 3~4의 밸류체인 분석은 LLM이 기업을 연결한다. 사람이 개입하는 플로우에서는 이상한 연결을 감지해 걸러낼 수 있지만, 자동매매에서는 그 기회가 없다. 한 번의 잘못된 연결이 주문으로 직행한다.

5. **초기 합의된 경계와의 일관성.** `PRD.md` Non-goals와 `CLAUDE.md` P1 원칙에 이미 매매 추천·가격 예측을 제외한 상태다. 자동매매는 그 경계의 가장 강한 형태이므로 같은 논리로 배제된다.

**재고 조건**: 아래 조건이 모두 충족된 시점에 재검토 가능하다.

- [ ] 시그널 백테스트 모듈이 구축되어 최소 1년치 과거 데이터로 점수 체계 유효성이 통계적으로 확인됨 (승률·평균 수익률·샤프 비율 등)
- [ ] 페이퍼 트레이딩 모드를 최소 3개월 돌려 실제 시그널 전략이 무의미하지 않음을 입증
- [ ] 포지션 사이징·손절 규칙·최대 손실 한도가 명시적으로 설계됨
- [ ] LLM 출력에 의존하지 않는 결정 규칙 경로가 확보됨 (할루시네이션이 주문으로 이어지지 않도록)

이 네 조건 중 하나라도 미충족 상태에서는 자동매매 논의를 재개하지 않는다.

**일자**: 2026-04-21

## 2. LLM 백엔드: Claude Code CLI (Max 플랜) 주 엔진 + Ollama 보조

**상황**: 분석·요약에 사용할 LLM 선택. Claude API, OpenAI, Ollama, 하이브리드 검토.

**결정**: Claude Code CLI 를 `subprocess.run(["claude", "-p", ...])` 로 호출해 Max 플랜 ($100/월) 할당량을 사용. 한도 초과 대비 Ollama 로컬 폴백.

**근거**:
- Max 플랜 이미 구독 중 → 추가 비용 $0
- `claude -p` CLI는 Max 플랜 청구를 지원하지만, `anthropic` Python SDK는 API 키 전용이라 Max 미지원
- 밸류체인 분석 같은 복잡한 추론에서 Claude 품질이 Ollama 로컬 모델 대비 체감 차이 큼
- 단순 반복 작업(용어 주석 사전 생성 등)은 Ollama로 비용 없이 처리 가능
- `ANTHROPIC_API_KEY` 환경 변수가 설정되면 Claude Code가 구독 대신 API 과금하는 함정이 있음 — 반드시 제거 필요

**재고 조건**:
- Anthropic Agent SDK가 Max 플랜 지원 시 subprocess wrapping 대신 SDK 직접 호출로 전환
- Max 플랜 한도 지속 초과 시 과금 API 전환 또는 Ollama 비중 확대 검토

**일자**: 2026-04-21

## 3. 관심 종목: 고정 리스트 대신 전체 시장 스캔 + 시그널 스코어링

**상황**: 수집 대상을 "100개 내 관심 종목 고정 리스트"로 할지, "전체 시장 스캔 후 시그널 기반 부상"으로 할지 결정.

**결정**: 전체 시장 스캔. DART 공시는 전체 받고, 시그널 점수가 임계값(60점)을 넘은 항목만 알림·브리핑에 노출.

**근거**:
- 사용자가 "100개+ 넓게 스캔" 방향 선호 명시
- 고정 리스트 방식은 새 테마·새 기업의 부상을 놓침
- DART API는 일 10,000 요청 한도로 전체 스캔에 여유 있음
- 필터링은 수집 이후 단계에서 수행해 효율적 (중복 API 호출 방지)

**재고 조건**: 전체 스캔 시 DART API 한도 초과가 반복되면 주요 시장(KOSPI 200 + KOSDAQ 150) 우선 스캔으로 축소.

**일자**: 2026-04-21

## 4. 카카오 알림 채널: 사업자 등록 없는 "나에게 보내기" API 사용

**상황**: 카카오톡 알림톡(사업자 전용)과 나에게 보내기 API(개인용) 중 선택.

**결정**: 개인 OAuth + "나에게 보내기" API (`/v2/api/talk/memo/default/send`).

**근거**:
- 사용자가 사업자 미등록, 개인 용도
- 알림톡은 비즈니스 채널 + 솔루션사 경유 필요 (건당 8~15원)
- "나에게 보내기"는 무료, 썸네일·링크·버튼 지원, 용도에 충분
- 토큰 관리는 refresh_token 자동 갱신 로직으로 해결

**제약**:
- 본인에게만 전송 가능 (타인에게는 알림톡 필요)
- 메시지 길이 4KB 제한 → Week 2 웹페이지 링크 방식으로 우회
- refresh_token 자체는 약 2개월 만료 → 주기적 재인증 알림 필요 (Week 2에 구현)

**재고 조건**: 타인 (가족 등) 에게도 브리핑 공유 필요성 발생 시 → 알림톡 사업자 등록 또는 텔레그램 봇 추가.

**일자**: 2026-04-21

## 5. 배포 방식: 로컬 macOS + launchd (클라우드 서버 미사용)

**상황**: 브리핑 시스템을 어디서 실행할지. 로컬 맥 vs VPS (AWS, Oracle Cloud 무료 티어) vs 컨테이너 호스팅.

**결정**: 로컬 macOS + launchd 스케줄링. 클라우드 서버 안 씀.

**근거**:
- 개인 사용자 1명, 24/7 uptime 불필요
- 맥이 켜져 있는 시간이 장 시간과 대체로 일치 (오전 9시 ~ 오후 11시)
- 클라우드 비용·관리 overhead 회피
- API 키·카카오 토큰 등 비밀 정보를 로컬에만 두는 프라이버시 이점
- launchd는 cron 대비 sleep 복구 안정적 (`RunAtLoad`, `StartCalendarInterval` 조합)

**제약**:
- 맥이 sleep 상태면 아침 브리핑이 늦어질 수 있음 → `pmset repeat wakeorpoweron` 로 자동 wake 설정 필요
- 맥이 완전히 꺼져있으면 해당 날 브리핑 실패 (다음 날 catch-up)

**재고 조건**: 해외 여행 등으로 장기간 맥이 꺼져 있게 되는 상황이 반복되면 → Oracle Cloud 무료 VPS 이관 고려 (ARM Ampere A1, 4 vCPU·24GB RAM 무료). 또는 아래 Claude Code Routines 재검토.

**검토했으나 기각된 대안: Claude Code Routines** (2026-04 research preview)

Anthropic이 제공하는 클라우드 스케줄 실행 기능. Max 플랜에 포함. 맥 꺼져있어도 실행되는 매력적 특성. 이 프로젝트는 아침 배치 전용이므로 "최소 주기 1시간" 제약은 걸림돌이 아니지만, **DB 영속성 문제로 현 시점 부적합**:

1. **환경이 매 실행마다 초기화** ← **결정적 제약**
   - 로컬 SQLite (`seen`, `llm_cache`, `glossary`) 영속성 없음
   - dedup 없으면 같은 공시 반복 알림, LLM 캐시 없으면 비용 폭증
   - 우회책 (매번 GitHub commit / 외부 DB / Vercel KV) 모두 로컬 SQLite보다 복잡
2. **GitHub repo 클론 방식** — `.kakao_tokens.json`, `.env` 같은 로컬 토큰 관리 패턴과 충돌
3. **Anthropic 클라우드 IP** — DART·카카오 API의 한국 IP 요구 가능성 (검증 필요)
4. **전체가 Claude Code agent 세션** — 단순 `claude -p` 호출이 아니라 매번 agent가 컨텍스트 로드 → 토큰 소비 폭증

**부분 적용 가능한 예외**: Week 4의 **주간 리포트 생성** (일요일 23:00, 상태 의존 낮음, 결과물을 PR로 push → Vercel 자동 배포) 만큼은 Routines에 적합할 수 있음. Week 4 착수 시점에 DB 의존성 재확인 후 결정.

**재고 트리거**:
- 해외 여행·장기 부재로 아침 브리핑 실패가 월 1회 이상 발생
- DB 영속성 우회 패턴 정립 (예: 매일 아침 `seen.db` 를 GitHub에 auto-commit)
- 그 시점에 launchd → Routines 또는 Oracle Cloud VPS 중 선택

**해외 여행·장기 부재 시 임시 전환 경로** (사전 준비용)

launchd 유지가 기본이되, **일시적으로 맥을 쓸 수 없는 기간**만 다음 경로로 전환하는 scheme을 미리 준비해둔다:

- **클라우드 스택**: GitHub Actions (cron 트리거) + Supabase (DB) + Anthropic API 키 (LLM)
- **LLM 비용**: Max 플랜 CLI는 ToS상 GitHub Actions에서 사용 불가 (2026-02 Anthropic 정책 참조). API 키로 전환 시 월 $10~30 예상
- **시간 지연**: GitHub Actions 스케줄 트리거는 평균 15~40분 지연. 정확한 06:00 도착 불가능. 여행 기간만 쓸 거라면 감수 가능
- **DB 동기화**: 출국 전 `seen.db`·`llm_cache.db`·`glossary.db` 를 Supabase로 export. 귀국 후 반대로 import
- **전환 액션**:
  - 출국 전: (1) GitHub Actions workflow enable (2) Supabase project ID를 repo secrets에 설정 (3) `ANTHROPIC_API_KEY` secret 설정 (4) launchd job 비활성화 `launchctl unload ...`
  - 귀국 후: 역순으로 복원

**준비 필요 사항 (Week 2 DoD 이후)**:
- `.github/workflows/morning.yml` workflow 파일 (평소엔 disabled 상태로 repo에 보관)
- `scripts/db_to_supabase.py` / `scripts/supabase_to_db.py` 동기화 헬퍼
- Anthropic Console에서 API 키 발급 (평소엔 미사용, 전환 시에만 활성)
- `config.py` 에 `USE_ANTHROPIC_API` 환경 변수 분기

이 준비는 Week 2 완료 후 하루 정도 시간 투자하면 충분. 실제 해외 출발 직전에 당황하지 않게 **문서화된 체크리스트**를 `scripts/travel-mode-setup.md` 에 따로 작성.

## 6. 다국어: 한국어 기본 + 영어 선택 (lazy generate)

**상황**: 다국어 범위와 LLM 비용 trade-off 결정.

**결정**:
- 기본 언어 한국어 (ko). 영어 (en) 는 사용자가 토글했을 때만 활성
- 파이프라인 기본 생성 언어는 **한국어만** (LLM 요약·주석 모두)
- 영어 모드 선택 시 **lazy하게** 해당 콘텐츠만 재생성 → DB에 `(content_hash, lang)` 로 캐시
- UI chrome (메뉴·버튼·섹션명) 은 정적 사전 파일 (`i18n/ko.json`, `i18n/en.json`)
- 뉴스 원문 제목은 **번역하지 않음** (소스 언어 유지)

**근거**:
- Max 플랜 할당량 보호: 모든 콘텐츠를 ko+en 이중 생성하면 LLM 호출 2배
- 사용자가 실제로 영어를 쓰는 경우는 드물 것으로 예상. Lazy가 비용·품질 면에서 합리적
- 원문 제목 번역은 역효과: 출처 찾기 어려워지고 번역 정확도 의심 여지. 언어 혼재를 유지하고 UI가 이를 지원

**재고 조건**: 사용 패턴이 en 모드 빈번 사용으로 수렴하면 eager 생성으로 전환 (모든 콘텐츠를 발행 시점에 ko+en 둘 다 생성).

**일자**: 2026-04-21

## 7. UI/UX 디자인 언어: 토스(Toss) 디자인 철학 기반 refined minimalism

**상황**: 프론트엔드 스택과 디자인 시스템 확정. 초기에 "Pretendard + warm dark + 점수 계층"으로 정했으나, 구체적 목업 검토 결과 일반적인 대시보드/info-dense 스타일이 되어 피드백을 받고 재정립.

**결정**: 토스(Toss) 디자인 철학 기반 refined minimalism. 5가지 핵심 원칙:

1. **Typography carries hierarchy, not chrome** — 뱃지·컬러 스트립·보더로 중요도를 표현하지 않고, 크기·굵기 대비만으로 위계를 만든다
2. **Conversational copy** — "주목도 85 · 긍정 해석" 같은 라벨 대신 "지금 가장 중요해요" 같은 말투
3. **Numbers as heroes** — 숫자가 핵심 정보일 때 22~28px / 700 으로 과감하게
4. **One focused action per card** — 버튼 3개 나열 대신 주요 CTA 1개 + 보조 2개
5. **Generous whitespace** — 카드 padding 22~28px, 섹션 간격 28~32px

**구체 적용**:
- 폰트: Pretendard Variable (유지), 굵기 대비 400/700 중심 (500 최소, 600 금지)
- 색상: 토스 그레이 팔레트 10단계 (`#F9FAFB` ~ `#191F28`). UI 99%가 그레이
- 시그널 색상: 배경·테두리 대신 **6px 원형 dot + 같은 색 레이블**로만 표현
- 주 CTA 버튼: 파랑(#3182F6) 대신 **거의 검정(#191F28)** 사용 (토스 관례)

- 한국 증시 관례 준수: 상승 빨강(#F04452), 하락 파랑(#3182F6)

**카피 가이드 병행 도입**:
- 반말·느낌표·한자어 남용 금지
- 상황별 표준 카피 정립 ("지금 가장 중요해요", "자세히 →", "전체 모두 보기 →" 등)

**상세**: [`docs/DESIGN.md`](DESIGN.md) 전면 개정 (v2).

**근거**:

## 8. 배포 형태: PWA (Progressive Web App) on Vercel

**상황**: 웹 프론트엔드 배포 형태 결정. 단순 정적 HTML 페이지 / PWA / React Native 네이티브 앱 / Capacitor 하이브리드 중 선택.

**결정**: **PWA (Progressive Web App)** + Vercel 무료 호스팅. 구조:

- **앱 셸 + 데이터 분리**: HTML/CSS/JS는 static build, 매일 브리핑 데이터는 JSON (`/api/briefing/2026-04-21.json` 또는 static file) 로 분리
- **Service Worker**: 앱 셸은 영구 캐시(Cache-First), 데이터는 Network-First + stale 2분 → 오프라인에서도 마지막 브리핑 열람 가능
- **manifest.json**: 이름·아이콘 세트(192/512px)·테마 색상·display: standalone
- **호스팅**: Vercel 무료 티어 (자동 HTTPS, GitHub 연동 자동 배포, 커스텀 도메인 무료)
- **프레임워크**: 정적 HTML + vanilla JS 조합 또는 Next.js 14 + `next-pwa`. 결정은 Week 2 착수 시점에 (복잡도 판단 후)

**근거**:
- **단일 코드베이스로 모바일·데스크탑 커버** — 네이티브·하이브리드 불필요
- **홈 화면 설치** — 매일 아침 사용 시나리오에 잘 맞음 (앱 아이콘 tap → 전체화면)
- **오프라인 캐싱** — 지하철·엘리베이터 등 신호 불안정 구간에서도 열람 가능
- **웹 푸시 알림** (Week 3+) — 카카오톡 보조 채널. iOS 16.4+ 는 PWA 설치 후 지원
- **비용 $0** — Vercel 개인 프로젝트 무료 티어로 충분
- **네이티브 앱 대비 배포 마찰 없음** — 스토어 심사·인증서 관리 불필요

**제약 사항**:
- iOS Safari 는 일부 PWA 기능에 제약 (예: 푸시 알림은 홈 화면 설치 후에만)
- 카카오톡 인앱 브라우저에서는 PWA 설치 UX 제한적 — 브라우저로 열기 유도 필요
- HTTPS 필수 → Vercel/Netlify/GitHub Pages 같은 자동 HTTPS 호스팅 필수

**재고 조건**:
- 웹 푸시 알림 품질이 카톡 대비 낮아 사용 경험 저하 시 → 카톡만 유지, PWA는 열람용으로만
- 사용 패턴이 데스크탑 위주로 수렴 시 → PWA 기능(설치·오프라인) 비중 낮춤
- 장기적으로 iOS 네이티브 기능(예: 위젯) 필요 시 → Capacitor 또는 네이티브 고려

## 9. 탭 구조: 시사 / 경제 2-탭 + 내부 국내/해외 필터

**상황**: 초기 설계 → 증권·공시 중심 단일 feed → 시사 분리 요구로 3탭(증권/시사/테마) → 빈 탭·용어 정확성 재검토를 거쳐 최종 2탭 확정.

**결정**: **2-탭 체계** (Segmented pill tab, 상단 고정) + **2차 국내/해외 필터** (각 탭 내부):

| 탭 | 콘텐츠 | 정렬 기준 | 국내/해외 필터 | 주 업데이트 |
|----|--------|----------|---------------|------------|
| **시사** (default, 좌측) | 정치·사회·국제·IT 뉴스 | **큐레이션** | 전체 / 국내 / 국제 | 매일 06:00 + 4시간마다 |
| **경제** (우측) | DART 공시 + 경제 뉴스 + SEC EDGAR + 테마 분석(Week 4+) | **시그널 점수** | 전체 / 국내 / 해외 | 매일 06:00 |

**"증권"이 아닌 "경제"로 정한 이유**:
- 실제 탭 콘텐츠가 공시뿐 아니라 경제 뉴스(한경·매경), 거시 지표(금리·환율), 해외 경제 이슈까지 포함 → "증권"은 좁음
- 일반 사용자 친화도: "증권"은 트레이더 어휘, "경제"는 누구에게나 친숙
- 포털·뉴스 사이트의 대분류가 전부 "경제"로 통일됨 (네이버 뉴스 카테고리, 다음 뉴스 등)

**"테마" 탭을 별도로 두지 않은 이유**:
- Week 2 완성 시점에 빈 탭이 됨 → "미완성" 인상
- 테마 기능 자체는 보존(Week 3~4 개발 예정) — UI 위치만 변경
- Week 4에 경제 탭 **상단 배너**로 추가 ("이번 주 주목 테마: 로봇·AI")
- 주간 리포트는 일요일 저녁 카톡으로 별도 URL 전송 (`/report/YYYY-Www`)

**기본 탭과 딥링크 전략**:
- PWA 앱 아이콘 / 북마크로 진입: **기본 = 시사 탭** (좌측)
- 아침 카톡 알림 클릭: **`?tab=economy` 딥링크 → 경제 탭으로 직접** (브리핑 핵심이 공시·시그널이라 바로 보여야 함)
- 긴급 알림(점수 85+): 공시 원문(DART URL) 직결이라 탭 무관

**국내/해외를 일급 네비게이션(탭)으로 삼지 않은 이유**:
- **관심사는 카테고리(시사·경제)이지 국가가 아님** — 삼성전자 공시와 NVIDIA 8-K를 시그널 점수 순으로 같이 보는 게 자연스러움
- **테마 분석은 국내외가 섞여야 의미** — 로봇 테마 = 국내 액추에이터 + 해외 AI칩
- **그래도 "국내만"/"해외만" 니즈는 실재** → 2차 필터로 해결

**필터 UI 원칙**:
- 스타일을 탭과 **의도적으로 다르게** (탭은 pill, 필터는 text+underline)
- 기본값: "전체"
- URL에 반영 (`?tab=economy&scope=all|domestic|foreign`)
- 필터 변경 시 스크롤 상단으로 리셋

**대시보드 느낌 방지**: 3-열 그리드 금지. 모든 탭이 **모바일 single-column 스크롤**. 데스크탑도 max-width 560px 중앙 정렬.

**연예·스포츠 제외**: 시사 탭에 정치·사회·국제·IT만. "데일리 브리핑" 성격상 의사결정·시야 확장에 도움 되는 이슈만 담음. 연예·스포츠는 포털 메인·SNS에서 자연스럽게 접하므로 본 앱이 중복해서 제공할 필요 없음.

**재고 조건**:
- 경제 탭 사용량이 시사 대비 압도적으로 높으면 → 기본 탭을 경제로 전환
- 테마 배너 사용량 높으면 별도 탭으로 승격
- 연예·스포츠 요구 지속 → 시사 탭 하단에 섹션 추가 (별도 탭은 금지)
- 탭 3개 이상 필요해지면 네비게이션 재설계 트리거

## 10. 카카오톡 메시지 간소화: text 템플릿 + 서비스명 + 링크

**상황**: 초기에 카톡 메시지를 feed 템플릿 + og 이미지 + 3줄 요약 본문 등으로 정교하게 설계. 사용자 피드백으로 "카톡은 링크용일 뿐 신경 안 써도 됨" 확인.

**결정**: **카카오 text 템플릿 단순 알림**으로 변경. 구조:
- 제목: `데일리 브리핑 · [날짜]` + 건수 한 줄
- 버튼: `열기`
- 이미지·feed 레이아웃 미사용

긴급 알림(점수 85+)만 유일한 예외 — 종목명 + 한 줄 요지 + `공시 원문` 버튼(DART URL 직결).

서비스명 기본값 **"데일리 브리핑"**. `config.py`의 `SERVICE_NAME` 상수로 분리.

**근거**:
- 카톡은 PWA로 유도하는 **접점**일 뿐. 정보의 종착지가 아님
- feed 템플릿·og 이미지 생성은 유지보수 비용만 높이고 사용자 가치 낮음
- 정보는 PWA 한 곳에만 있는 게 일관성 있음 (카톡에 있으면 "어느 쪽이 최신이지?" 혼란)
- text 템플릿은 발송 실패율 낮고 미리보기 안정적

**제거되는 작업 (로드맵에서 빠짐)**:
- og 이미지 생성 파이프라인 (Pillow 또는 Edge Function)
- feed 템플릿 JSON 구조 설계
- 본문 200자 압축 로직

**재고 조건**:
- 카톡 알림 클릭률이 저조해 사용자가 브리핑을 놓치는 패턴 발생 시 → 제목 카피 A/B 테스트 또는 하루 건수 노출

## 11. 용어 해설을 "부가 기능"에서 "핵심 기능"으로 격상

**상황**: 초기 설계에서 용어 주석(glossary)은 카드 하단 작은 `💡` chip 형태로 배치. 사용자 피드백: 초심자에게 용어 해설이 중요한데 UI에서 덜 보임.

**결정**:
- **용어 해설 UI를 카드 본문의 primary 요소로 격상**
- **히어로 카드**: inset 해설 박스를 기본 펼쳐둠 (접힘 상태 아님)
- **일반 카드**: chip + inset 펼침 UI를 본문 바로 아래에 배치 (구분선 위)
- **첫 방문 사용자**: 첫 3건 카드의 용어 해설 자동 펼침
- **"이미 안다" 옵션**: 사용자가 특정 용어를 탭하면 "알겠어요" 버튼 노출 → localStorage 저장
- **설정 전역 토글**: "용어 해설 기본 펼침" on/off (기본값 on)
- **증권뿐 아니라 시사 탭에도 동일 적용** — "원내대표", "대법원 전원합의체", "추경" 같은 시사 용어

**근거**:
- 이 프로덕트의 차별화 가치는 "AI 요약"이 아니라 **"모르는 용어를 검색 없이 이해"**. 이게 안 보이면 차별점 사라짐
- 하단 작은 링크는 "부가 기능" 인상 → 초심자는 존재 자체를 모름
- Inline 펼침은 페이지 이동 없이 이해 가능, 읽기 흐름 유지
- "이미 안다" 옵션은 경험자에게도 UX 저하 없음

**구현 우선순위**:
- Week 2 Week 증권 탭 카드 컴포넌트 구현 시 즉시 적용
- 시사 탭 용어 해설 생성은 Week 3 (F32)

**재고 조건**:
- 용어 해설 "알겠어요" 클릭률이 극도로 높아 모든 사용자가 금세 접음 → 기본값 off로 전환
- 반대로 거의 아무도 접지 않으면 "상급자 모드" 설정 삭제
