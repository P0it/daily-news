# 공시 본문 분석 단계 설계

**작성일**: 2026-07-09
**상태**: 승인됨 (구현 계획 대기)

## 1. 문제

파이프라인이 공시의 **제목과 메타데이터만** 다룬다. 본문을 한 번도 열지 않는다.

- `collectors/dart.py` 는 `list.json` 만 호출한다. `report_nm`·`rcept_no`·회사명만 얻고
  본문(`document.xml`)은 받지 않는다.
- `collectors/edgar.py` 는 Atom 피드의 제목과 `Item 2.02` 같은 아이템 번호만 정규식으로
  추출한다. 8-K 에 첨부된 실적 보도자료(`EX-99.1`)는 열지 않는다.
- 그 결과 `analysis/scoring.py` 는 `"영업(잠정)실적" → 85점, mixed` 로 끝난다.
  매출이 컨센서스를 넘겼는지 밑돌았는지 시스템은 알지 못한다.
  `"합병" → 80점, mixed` 도 인수가·프리미엄·대가 형태를 모른다.
- `analysis/pick_verify.py` 의 2차 LLM 검증조차 "그 제목이 근거 목록에 실재하는가"만
  대조한다. 내용 해석이 아니라 존재 확인이다.

`mixed` 라는 방향값은 곧 "호재인지 악재인지 판단을 포기했다"는 뜻이다. 같은 실적 공시라도
서프라이즈 +12% 와 −12% 는 정반대 신호인데 현재는 둘 다 85점 `mixed` 로 동일하게 취급되어
picks 상위를 차지한다.

## 2. 목표와 비목표

**목표**

- 실적·합병 공시의 본문에서 정량 사실을 추출해 구조화한다.
- 그 사실로 `score` 와 `direction` 을 **규칙이** 재산정한다.
- 컨센서스에 부합한(정보량 없는) 실적 공시를 picks 상위에서 강등시킨다.

**비목표**

- 모든 공시 유형의 본문 분석. 실적·합병 두 유형으로 시작한다.
- LLM 의 자유 서술형 투자 의견. LLM 은 추출기로만 쓴다 (해설은 §7 의 후속 단계).
- 새 데이터 소스 유료 구독. yfinance·DART Open API·SEC EDGAR 무료 범위 안에서 해결한다.

## 3. 접근

**A(정량 추출) → C(추출 후 해설)** 순서로 간다. 먼저 숫자를 뽑아 규칙으로 판단하고
(A), 그 위에 추출된 숫자에만 근거한 짧은 해석을 얹는다(C). B(LLM 해설 단독)는 채택하지
않는다 — 할루시네이션에 무방비이고 `picks_outcomes.py` 로 성과를 백테스트할 수 없다.

**실적 판단 기준은 시장별로 다르게 잡는다.**

- **해외**: 컨센서스 대비 서프라이즈 %.
- **국내**: 전년동기 대비(YoY) 및 흑자/적자 전환.

국내 개별종목 컨센서스는 무료로 신뢰성 있게 얻을 소스가 사실상 없다. 국내에도 서프라이즈를
강제하면 대부분이 결측으로 빠진다. 반면 해외는 yfinance 가 커버하고, 시스템 소유자의 관심도
해외에 있으므로 신호 품질이 높은 쪽에 투자한다.

### 검증된 가정

설계 전 실측으로 확인했다.

- `yfinance.Ticker(t).earnings_dates` 가 `EPS Estimate`·`Reported EPS`·`Surprise(%)` 를
  무료로 반환한다. **해외 실적은 8-K 본문 파싱도 LLM 호출도 없이 사실을 확보한다.**
- DART `document.xml` 은 zip 을 반환하며, 압축을 풀면 XML 본문이 나온다
  (샘플 23,753자). 태그 제거 후 텍스트로 쓸 수 있다.
- DART 는 10일 창에 6,573건이 유입되지만 그 중 `실적` 공시는 4건이었다. 전건 본문 수집은
  불가능하고 불필요하다 — **게이팅이 필수이며, 실무 부하는 하루 수 건 수준이다.**

## 4. 아키텍처

새 단계는 **수집과 점수 산정 사이**에 들어간다. 현재 `orchestrator.py` 의 3단계
(`score_report(it.title)` 로 끝나는 지점)를 5단계로 늘린다.

```
수집 → ① 제목 점수(기존) → ② 게이팅 → ③ 본문 수집 → ④ 사실 추출 → ⑤ 재점수 → picks
```

모듈은 수집 두 개, 추출 하나, 판단 하나로 나눈다. 셋은 서로 다른 이유로 실패하고 서로 다르게
테스트되기 때문이다.

| 모듈 | 책임 | 의존 |
|---|---|---|
| `collectors/filing_body.py` | DART `document.xml` zip 해제·태그 제거, EDGAR 8-K 의 `EX-99.1` 본문 | 네트워크만 |
| `collectors/earnings_surprise.py` | 티커 → 최근 분기 EPS 추정·실적·서프라이즈% | yfinance |
| `analysis/filing_facts.py` | 본문 → `FilingFacts` 구조체 | LLM (추출 전용) |
| `analysis/filing_score.py` | `FilingFacts` → `(score, direction)` | **없음 — 순수 함수** |

`filing_score.py` 가 순수 함수라는 점이 설계의 중심이다. 판단 로직 전체를 네트워크·LLM 없이
픽스처만으로 테스트할 수 있고, 임계값 변경의 영향을 백테스트로 측정할 수 있다.

### 게이팅 규칙

본문을 여는 조건은 셋을 모두 만족할 때다.

1. 공시 유형이 `earnings` 또는 `merger` 로 분류됨 (제목 키워드 / 8-K Item 코드)
2. 제목 점수가 임계값 이상 (기본 70)
3. 위 조건을 통과한 것 중 점수 상위 N건 (기본 25)

N=25 는 하루 LLM 호출 25회에 해당한다. 해외 실적은 LLM 을 쓰지 않으므로 실제 호출은 이보다
적다.

## 5. 데이터 모델

`FilingFacts` 는 유형별 필드를 갖는다. 모든 필드는 optional — 결측은 `None` 으로 둔다.

**실적 (earnings)**

- 해외: `eps_estimate`, `eps_reported`, `surprise_pct`
- 국내: `revenue`, `operating_income`, `net_income`, 각각의 `*_yoy_pct`,
  `turnaround` (`"흑자전환"` | `"적자전환"` | `None`)

**합병·인수 (merger)**

- `acquirer`, `target`, `deal_value`, `premium_pct`,
  `consideration` (`"cash"` | `"stock"` | `"mixed"`)

추출한 사실은 `item.extra["facts"]` 에 스탬프해 `hot_issues`·`picks`·UI 가 함께 쓴다.

## 6. 재점수 규칙

판정 순서가 중요하다. **서프라이즈 → 전환 → YoY** 순으로 본다. 컨센서스 대비가 있으면 YoY 는
보지 않는다(YoY 에는 시장이 이미 알던 정보가 섞여 있다). 전환은 연속 지표로 표현되지 않는
이산 사건이므로 YoY 보다 먼저 판정한다.

| 사실 | 점수 | 방향 |
|---|---|---|
| 서프라이즈 ≥ +10% | 88 | positive |
| 서프라이즈 +5 ~ +10% | 80 | positive |
| 서프라이즈 −5 ~ +5% | **55** | neutral |
| 서프라이즈 −10 ~ −5% | 80 | negative |
| 서프라이즈 ≤ −10% | 88 | negative |
| 적자전환 | 92 | negative |
| 흑자전환 | 90 | positive |
| 영업이익 YoY ≥ +30% | 85 | positive |
| 영업이익 YoY +10 ~ +30% | 75 | positive |
| 영업이익 YoY −10 ~ +10% | **55** | neutral |
| 영업이익 YoY −30 ~ −10% | 75 | negative |
| 영업이익 YoY ≤ −30% | 85 | negative |
| 피인수 + 프리미엄 ≥ 30% | 90 | positive |
| 피인수 + 프리미엄 10 ~ 30% | 82 | positive |
| 피인수 + 프리미엄 미상 또는 < 10% | 75 | positive |
| 인수 주체 | 70 | mixed |

인수 주체를 `mixed` 로 두는 이유는 주가 영향이 양방향이기 때문이다(시너지 vs 고가 인수).
피인수 대상은 프리미엄이 확인되지 않아도 호재이므로 `positive` 로 두되 점수를 낮춘다.

핵심은 **`−5 ~ +5%` 구간의 55점 강등**이다. 현재 이런 공시도 85점을 받아 picks 상위를
잠식한다. 컨센서스에 부합한 실적은 새 정보가 없다는 뜻이므로 내려가는 것이 맞다.
이 단계의 진짜 값어치는 새 픽을 더하는 것보다 **가짜 픽을 걷어내는 데** 있다.

**임계값 근거와 재고 조건**: 위 수치(±5%/±10%, YoY ±30%, 프리미엄 30%)는 업계 통용
관행에 따른 초기값이며 실측 근거는 아직 없다. `picks_outcomes.py` 로 구간별 실제 수익률을
집계한 뒤 조정한다. `CLAUDE.md` 의 "알림 임계값 임의 변경 금지" 조항에 따라, 향후 변경 시
사용자 승인과 `DECISIONS.md` 기록을 거친다.

**폴백**: 추출 실패 시 기존 제목 점수를 그대로 쓴다. 이 단계는 최악의 경우에도 현행과
동일하며, 결코 더 나빠지지 않는다.

## 7. 후속 단계 (C)

`FilingFacts` 가 채워진 픽에 한해, **추출된 숫자만을 근거로** 한 문장 해석을 생성한다.
프롬프트에 본문 원문을 넣지 않고 구조화된 사실만 전달해 그라운딩을 강제한다.
A 가 안정화된 뒤 별도 명세로 진행한다.

## 8. 에러 처리

`CLAUDE.md` 원칙을 따른다.

- 공시별 본문 수집과 추출을 개별 `try/except` 로 감싼다. 한 건 실패가 나머지를 막지 않는다.
- LLM 추출기는 JSON 스키마를 강제한다. 파싱 실패는 `None` 반환 후 제목 점수로 폴백.
- 모든 외부 호출에 timeout: 본문 수집 15초, LLM 45초.

### 캐싱 — 기존 `llm_cache` 재사용, 새 테이블 없음

Supabase 에 `llm_cache(content_hash PK, task, output, model, created_at)` 가 이미 있고
현재 미사용(0행)이다. 이를 그대로 쓴다.

- `content_hash` = `sha256("filing_facts:{rcept_no|accession}")`
- `task` = `"filing_facts"`
- `output` = `FilingFacts` 직렬화 JSON

**추출 결과를 캐싱하므로 재실행 시 본문 수집과 LLM 호출을 둘 다 건너뛴다.** 원본 본문
(샘플 23KB XML)을 DB 에 적재할 이유가 없어 별도 캐싱 테이블을 만들지 않는다.

### 스키마 변경 — `pick_outcomes.facts_json`

`pick_outcomes` 에는 `direction`·`signal`·`alpha_*` 성과 컬럼이 있으나 추출한 사실
(`surprise_pct` 등)을 담을 자리가 없다. 저장하지 않으면 §6 의 "구간별 실제 수익률 집계 후
임계값 조정"이라는 재고 조건을 이행할 수 없다.

`facts_json text` (nullable) 컬럼 하나를 추가한다. 기존 175행에 영향이 없고 nullable 이라
롤백이 안전하다. `CLAUDE.md` P4 에 따라 사용자 확인 후, 코드 마이그레이션 스크립트로 적용한다.

LLM 은 **본문에 있는 숫자를 옮겨적는 일만** 한다. 해석도 추론도 하지 않으므로
`pick_verify.py` 에서 겪은 할루시네이션의 여지가 구조적으로 없다. 추출된 수치가 본문에
문자열로 실재하는지 대조하는 검증을 추가한다.

## 9. 테스트

- **단위**: `filing_score.py` 는 순수 함수이므로 사실 픽스처만으로 전 규칙을 커버한다.
  경계값(±5%, ±10%, ±30%)을 명시적으로 테스트한다.
- **파서**: 저장한 실제 DART XML / EDGAR HTML 픽스처로 테스트한다.
- **통합**: 네트워크 실호출은 `@pytest.mark.integration` 으로 분리해 기본 실행에서 제외한다.
- **LLM**: 프롬프트 구조와 JSON 후처리만 검증한다 (비결정성 때문).
